Управление каталогами¶

Каталог поддерживает записи метаданных, состоящие из фрагментов метаданных. Идея состоит в том, что фрагменты метаданных могут использоваться более чем в одной записи метаданных.

Вот типичный пример фрагмента. Это ответственная сторона, и она может использоваться в одной и той же записи метаданных более одного раза или более чем в одной записи метаданных, если применимо.

<gmd:CI_ResponsibleParty xmlns:gmd="http://www.isotc211.org/2005/gmd" xmlns:gco="http://www.isotc211.org/2005/gco" >
  <gmd:individualName>
    <gco:CharacterString>John D'Ath</gco:CharacterString>
  </gmd:individualName>
  <gmd:organisationName>
    <gco:CharacterString>Mulligan &amp; Sons, Funeral Directors</gco:CharacterString>
  </gmd:organisationName>
  <gmd:positionName>
    <gco:CharacterString>Undertaker</gco:CharacterString>
  </gmd:positionName>
  <gmd:role>
    <gmd:CI_RoleCode codeList="./resources/codeList.xml#CI_RoleCode" codeListValue="pointOfContact"/>
  </gmd:role>
</gmd:CI_ResponsibleParty>

Фрагменты метаданных, сохраняемые в базе данных каталога, называются подшаблонами. Это в основном по историческим причинам, поскольку подшаблон подобен записи метаданных шаблона в том смысле, что его можно использовать в качестве «шаблона» для создания новой записи метаданных.

Фрагменты могут быть вставлены в запись метаданных в 2-х режимах:

  • путем копирования/вставки

  • по ссылке (если включена поддержка xlink. См. Метаданные XLink)

При использовании XLink, если фрагмент обновляется, то соответствующий фрагмент во всех записях метаданных также будет обновлен (проверьте кэш XLink).

Фрагменты можно создавать путем сбора (см. Сбор фрагментов метаданных для поддержки повторного использования) или импортировать с помощью страницы импорта метаданных.

В этом разделе руководства описываются:

  • как управлять каталогами подшаблонов

  • как извлечь фрагменты из существующего набора записей метаданных и сохранить их как подшаблоны

  • как управлять кешем фрагментов, используемым для ускорения доступа к фрагментам, которых нет в локальном каталоге

Управление каталогами подшаблонов¶

Существуют некоторые различия между обработкой подшаблонов и записей метаданных. В отличие от записей метаданных, подшаблоны не имеют согласованного корневого элемента, используемая ими схема метаданных может быть неузнаваема, они не отображаются в основных результатах поиска (если только они не являются частью записи метаданных). Таким образом, панель редактора позволяет вам искать и управлять привилегиями для записей каталога.

На панели редактора выберите «Управление каталогом», чтобы получить доступ к редактору записей каталога:

Если вы не видите вкладку «Организации и контакты», убедитесь, что вы создали подшаблоны для контактов для своего профиля метаданных и загрузили их с помощью раздела «Метаданные и шаблоны».

На этой странице редакторы могут выбирать тип каталога с помощью верхних вкладок, редактировать/удалять/импортировать новые подшаблоны.

Чтобы импортировать новые записи, используйте страницу импорта метаданных и выберите соответствующий тип записи:

Как и записи метаданных, им присваивается целочисленный идентификатор, и они хранятся в таблице метаданных каталога (с полем шаблона, установленным в «y»).

Вставьте запись каталога в запись метаданных¶

В редакторе метаданных каталог можно использовать, например, для заполнения контактов.

Откройте селектор каталогов, выберите контакт, а затем выберите роль контакта.

Извлечение подшаблонов из записей метаданных¶

Многие сайты имеют существующие записи метаданных с общей информацией, например. контактная информация в элементе ISO CI_Contact. Подобные записи каталога можно извлечь из выбранного набора записей метаданных с помощью API «Извлечение подшаблонов».

Для использования этой функции необходимо выполнить следующий набор шагов:

  • Убедитесь, что вы понимаете, что такое XPath — см., например, http://www.w3schools.com/xpath/default.asp.

  • Определите фрагменты метаданных, которыми они хотели бы управлять как многократно используемые подшаблоны в записи метаданных. Это можно сделать с помощью XPath. например. XPath .//gmd:CI_ResponsibleParty идентифицирует все ответственные стороны в записях. Пример такого фрагмента (взятого из одной из выборочных записей) показан в следующем примере:

<gmd:CI_ResponsibleParty>
   <gmd:individualName>
      <gco:CharacterString>Jippe Hoogeveen</gco:CharacterString>
   </gmd:individualName>
   <gmd:organisationName>
      <gco:CharacterString>FAO - NRCW</gco:CharacterString>
   </gmd:organisationName>
   <gmd:positionName>
      <gco:CharacterString>Technical Officer</gco:CharacterString>
   </gmd:positionName>
   <gmd:contactInfo>
      <gmd:CI_Contact>
         <gmd:phone>
            <gmd:CI_Telephone>
               <gmd:voice gco:nilReason="missing">
                  <gco:CharacterString/>
               </gmd:voice>
               <gmd:facsimile gco:nilReason="missing">
                  <gco:CharacterString/>
               </gmd:facsimile>
            </gmd:CI_Telephone>
         </gmd:phone>
         <gmd:address>
            <gmd:CI_Address>
               <gmd:deliveryPoint>
                  <gco:CharacterString>Viale delle Terme di Caracalla</gco:CharacterString>
               </gmd:deliveryPoint>
               <gmd:city>
                  <gco:CharacterString>Rome</gco:CharacterString>
               </gmd:city>
               <gmd:administrativeArea gco:nilReason="missing">
                  <gco:CharacterString/>
               </gmd:administrativeArea>
               <gmd:postalCode>
                  <gco:CharacterString>00153</gco:CharacterString>
               </gmd:postalCode>
               <gmd:country>
                  <gco:CharacterString>Italy</gco:CharacterString>
               </gmd:country>
               <gmd:electronicMailAddress>
                  <gco:CharacterString>jippe.hoogeveen@fao.org</gco:CharacterString>
               </gmd:electronicMailAddress>
            </gmd:CI_Address>
         </gmd:address>
      </gmd:CI_Contact>
   </gmd:contactInfo>
   <gmd:role>
      <gmd:CI_RoleCode codeList="http://standards.iso.org/ittf/PubliclyAvailableStandards/ISO_19139_Schemas/resources/codelist/ML_gmxCodelists.xml#CI_RoleCode"
                       codeListValue="pointOfContact"/>
   </gmd:role>
</gmd:CI_ResponsibleParty>

  • Определите и запишите XPath поля или полей внутри фрагмента, текстовое содержимое которого будет использоваться в качестве идентификатора подшаблона. Этот XPath должен относиться к корневому элементу фрагмента, указанному на предыдущем шаге. Так, например, в приведенном выше фрагменте мы могли бы выбрать .//gmd:electronicMailAddress/gco:CharacterString/text() в качестве идентификатора для создаваемых фрагментов.

  • На странице API выбираем операцию registers/collect:

  • Заполните форму данными, собранными на предыдущих шагах.

  • Извлеченные подшаблоны можно предварительно просмотреть в режиме GET, а после проверки использовать метод PUT для сохранения результатов в каталоге.

Наконец, перейдите в интерфейс управления каталогом подшаблонов, и вы сможете выбрать корневой элемент ваших подшаблонов, чтобы изучить извлеченные подшаблоны.

Индексация подшаблона основана на схеме (подробности см. в папке index-fields). В настоящее время подшаблоны индекса ISO19139 используются в качестве корневого элемента:

  • gmd:CI_ResponsibleParty

  • gmd:MD_Distribution

  • gmd:CI_OnlineResource

  • gmd:EX_Extent

Синхронизация подшаблонов с записями метаданных¶

После создания каталог предоставляет возможность синхронизировать записи метаданных с записями каталога. Для этого воспользуйтесь страницей тестирования API.

Процесс синхронизации использует те же параметры, что и процесс сбора, с двумя необязательными аргументами:

  • propertiesToCopy to preserve some element which may be defined in the fragment in the metadata (eg. contact role)

  • substituteAsXLink to indicate if copy/paste mode or XLink mode should be used.

Управление кэшем фрагментов¶

Если записи метаданных в вашем каталоге ссылаются на фрагменты с внешних сайтов, каталог кэширует эти фрагменты после первого поиска, чтобы уменьшить объем сетевого трафика и ускорить отображение записей метаданных в результатах поиска.

Кэш обрабатывается автоматически с помощью Java Cache System (JCS). JCS разумно обрабатывает большие кэши:

  • определение максимального количества кэшируемых объектов

  • использование как можно большего объема оперативной памяти перед перемещением во вторичное хранилище (диск)

  • обеспечение постоянства кеша: кеш сохраняется на диск при завершении работы веб-приложения и восстанавливает кеш с диска при перезапуске

Вы можете настроить параметры JCS в GeoNetwork, используя файл конфигурации JCS в INSTALL_DIR/web/geonetwork/WEB-INF/classes/cache.ccf.

Некоторые операции в каталоге (например, сбор), которые создают фрагменты метаданных, автоматически обновляют кэш XLink при создании нового фрагмента. Однако, если вы линкуете фрагменты с внешнего сайта, то в зависимости от того, как часто происходит изменение, вам нужно будет вручную обновлять кэш XLink. Для этого вам необходимо перейти на страницу «Администрирование» и выбрать функцию «Очистить кэш XLink и перестроить индекс записей с помощью XLink», как показано на следующем снимке экрана страницы «Администрирование».
